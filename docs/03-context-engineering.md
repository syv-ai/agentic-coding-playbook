# Module 03: Context Engineering Fundamentals

## The Core Skill

If you learn one thing from this workshop, make it this: the quality of what an agent produces is determined almost entirely by the context it has access to [^1]. Not the model. Not the tool. The context.

Context engineering is the practice of curating the information available to a language model during inference [^1]. It goes well beyond writing a good prompt. It's about managing the entire information landscape: system instructions, tool definitions, file contents, conversation history, and external data. The goal is to find the smallest set of high-signal tokens that maximize the likelihood of the outcome you want [^1].

This is the skill that separates developers who get reliable, consistent results from agents and developers who get expensive unpredictability [^2].

## What Is Context, Exactly?

When an agent processes your request, its "context window" holds everything it can see [^3]:

| What | When It Loads | Cost |
|------|--------------|------|
| System instructions | Every request | Constant |
| Your instruction file (CLAUDE.md, AGENTS.md, etc.) | Session start | Constant, every request |
| Tool definitions (names and descriptions) | Session start | Scales with number of tools |
| Conversation history | Accumulated over session | Grows with each exchange |
| File contents the agent reads | On demand | Per file, when read |
| Command output | On demand | Per command, when run |
| Loaded skills or plugins | On demand or session start | Varies |

Context is finite. Every token the agent spends attending to one piece of information reduces its ability to focus on everything else [^1]. This isn't a metaphor. Transformer architecture requires each token to attend to every other token, and accuracy measurably decreases as context grows. The literature calls this "context rot" [^1].

This has practical implications: you want to be strategic about what goes in.

## The Instruction File in Depth

Your AGENTS.md or CLAUDE.md is the most important piece of context you control. It loads every single request, so it's always competing for attention. Keep it lean and high-signal [^3][^4].

### What Belongs in the Instruction File

Things the agent should always know, regardless of what task you're working on:

```markdown
# Project: my-api

## Tech Stack
Go 1.23, PostgreSQL 16, Chi router, sqlc for queries

## Commands
- Build: `go build ./...`
- Test: `go test ./...`
- Lint: `golangci-lint run`
- Generate SQL types: `sqlc generate`

## Architecture
- cmd/server/main.go is the entry point
- internal/api/ contains HTTP handlers
- internal/store/ contains database queries (generated by sqlc)
- internal/domain/ contains business logic, no dependencies on HTTP or DB

## Conventions
- Errors wrap with fmt.Errorf("context: %w", err), never bare returns
- Handler functions take (w http.ResponseWriter, r *http.Request), no custom context types
- Database queries live in .sql files in internal/store/queries/, sqlc generates the Go code
- Tests use testcontainers for database tests, no mocks for the DB layer
```

Notice what's not in there: no explanations of why these conventions exist, no history of how the project evolved, no prose paragraphs about the team's philosophy. Just facts the agent needs to follow your conventions.

### What Does Not Belong

Anything the agent can derive by reading the code. If your function signatures are clear, you don't need to document them in the instruction file. If your test patterns are consistent, you don't need to describe them, the agent will read an existing test and match the style.

Also avoid:

**Long reference documentation.** If the agent sometimes needs your API style guide, make it a skill or a separate file it can load on demand, not something that eats context every single request [^3].

**Things that change often.** Current sprint goals, in-progress tickets, recent decisions. Those belong in your issue tracker or a separate doc, not in an always-loaded file.

**Instruction bloat.** Frontier models can follow roughly 150 to 200 instructions with reasonable consistency [^4]. Your tool's built-in system prompt already uses around 50 of those slots. Every line you add competes for that budget.

### The Feedback-Driven Approach

The best instruction files aren't written upfront. They evolve. Anthropic's recommendation for Claude Code [^3], and it applies to any tool:

| When This Happens | Do This |
|-------------------|---------|
| Agent gets a convention wrong twice | Add it to your instruction file |
| You keep typing the same prompt to start a task | Save it as a reusable skill or command |
| You paste the same procedure into chat for the third time | Capture it as a skill |
| Agent can't access data it needs | Connect that system via MCP or a tool |

Start minimal. Add when you observe failures. Remove when rules no longer apply. This is the "start minimal, evolve incrementally" principle from Module 00.

## Beyond the Instruction File: Organizing Context

As your setup grows, you need structure. Most tools support splitting instructions across multiple files and scoping them to specific situations.

### Rules and Scoped Instructions

Instead of one giant instruction file, you can organize by concern:

```
.claude/rules/
  typescript.md      # TypeScript conventions, only loaded when working on .ts files
  testing.md         # Test patterns
  api-design.md      # REST API conventions
```

Or in Cursor:

```
.cursor/rules/
  general.mdc
  react-components.mdc
  database.mdc
```

Scoped rules only load when the agent is working with matching files. This saves context for everything else [^3].

### Skills: On-Demand Knowledge

Skills (or their equivalent in other tools) are reusable instruction sets that load only when needed [^3]. They're the answer to "I want the agent to know about X, but not all the time."

Two types:

**Reference skills** provide knowledge the agent uses when relevant. Your API documentation, a style guide, framework patterns. These load when the agent determines they're relevant to the current task, or when you invoke them.

**Action skills** tell the agent to do something specific. A deployment checklist, a code review procedure, a database migration workflow. You invoke these explicitly.

The key insight: skill descriptions are what the agent sees at session start. The full content only loads when a skill is actually used [^3]. That means you can have dozens of skills without paying the context cost until you need one.

## The Altitude Problem

One of the most common mistakes in writing agent instructions is getting the abstraction level wrong [^1]. There are two failure modes:

**Too specific.** You hardcode exact steps, file paths, and implementation details. This is brittle. It breaks when anything changes, and it prevents the agent from using its judgment.

```markdown
# Too specific
When fixing a bug in src/api/users.ts, first check line 42 for the null check,
then verify the TypeORM query on line 87, then run npm test -- --grep "users".
```

**Too vague.** You assume shared context that the agent doesn't have. This leads to the agent guessing, usually wrong.

```markdown
# Too vague
Follow our coding standards and best practices.
```

The right level is somewhere in between. Give enough specifics that the agent knows what you mean, but frame them as principles the agent can apply to novel situations:

```markdown
# Right level
- API handlers validate input at the boundary, then call domain functions that assume valid data
- Database queries go through the repository layer, never called directly from handlers
- Error responses use RFC 7807 Problem Details format (see src/api/errors.go for the helper)
```

## Just-In-Time Context

Not everything needs to be pre-loaded. Modern agents use a hybrid approach [^1]: some context is loaded upfront (your instruction file, tool names), and the rest is retrieved on demand through tools.

This mirrors how humans work. You don't memorize an entire codebase before starting a task. You know the broad structure, and you look things up as you need them. Agents work the same way: they use file names, folder structure, and search to discover what they need [^1].

This means your project's organization is itself a form of context engineering. Clear naming conventions, consistent folder structure, and well-placed README files all help the agent navigate efficiently.

## Exercises

1. Audit your current instruction file (or create one). Is anything in there that the agent could infer from reading the code? Remove it.
2. Find one convention that you've had to correct the agent on more than once. Add it to the instruction file.
3. Identify one piece of reference documentation you keep pasting into conversations. Extract it into a separate file or skill.
4. Check the line count of your instruction file. If it's over 200 lines, look for content that can be moved to scoped rules or skills.
